# 大耳朵TV写作风格蒸馏 v0.5：ASR辅助版

> **结论先说**：v0.4 的判断基本成立，但 ASR 后要补一条关键修正：大耳朵TV不是“先完整说结论再展开”的平面文章逻辑，而是更像 **冷开场演示/吐槽 → 今天讲什么 → 真实使用依据 → 首先/第二/第三 → 但是 → 所以 → 购买/升级/行动建议** 的口播链路。  
> **当前可信度**：本版有 20 条视频、58 个片段、约 84.9 分钟 ASR 辅助；但未做 Claude 二次审阅，且 Whisper base 有错字，所以本版是 `ASR-assisted preliminary`，不能当逐字模仿稿。

## 1. 这版比 v0.4 多了什么

v0.4 基于标题、简介、元数据和 Claude 审阅，主要能判断：

```text
真实使用 / 强结论 / 优缺点 / 谁该买谁别买
```

v0.5 补上的，是口播层面的几个信号：

1. **冷开场比想象中重要**：很多视频不是直接“大家好”，而是先放一个产品镜头、体验片段、吐槽、荒唐场景或冲突点，再进入“今天跟大家分享”。
2. **“但是 / 所以”是核心节拍器**：ASR 样本里 `所以` 约 100 次，`但是` 约 95 次；它不是单纯口头禅，而是在完成“营销卖点 → 真实体验 → 消费建议”的转折。
3. **评测不是参数顺序，而是体验顺序**：先说“用了多久/为什么要讲”，再按体验点拆，参数只是支撑。
4. **缺点通常被放在信任节点**：不是一上来否定，而是先承认优点，再说“但是……”“唯一比较大的缺点……”“如果你是……”。
5. **VLOG 的结构更像小型故事**：目标/挑战/奇怪动机 → 过程受苦 → 观察/吐槽 → 结果悲剧/反转。

## 3. ASR 关键词信号

以下不是“语言风格精确统计”，而是从 58 个 ASR 片段里看口播结构的节拍信号：

| 词/结构 | ASR 命中 | 说明 |
|---|---:|---|
| 所以 | ~100 | 把体验压缩成建议，是最重要的收束词 |
| 但是 | ~95 | 从卖点转入限制、真实体验、缺点 |
| 如果 | ~27 | 面向不同人群做条件建议 |
| 今天 | ~25 | 开场任务声明 |
| 第二 | ~21 | 列表化拆解明显 |
| 首先 | ~18 | 主体段落启动器 |
| 第一 | ~17 | 条目化评测/推荐 |
| 体验 | ~16 | 频道核心词，不只讲参数 |
| 问题 | ~12 | 用问题推进判断 |
| 总之 | ~8 | 快速收束，不做长篇文学化结尾 |
| 优点 | ~7 | 常与“缺点”成对出现 |
| 缺点 | ~5 | 频率不高，但位置关键 |

**解释**：大耳朵TV的可迁移风格不是“多用口头词”，而是：

```text
我看到了一个卖点
  → 但是实际用会遇到 X
  → 所以你应该/不应该这样选
```

这比单纯写“优点很多、缺点也有”更有决策感。

## 4. 口播结构：真正可借鉴的节奏

### 4.1 冷开场：先给一个“发生了什么”的画面

ASR 显示，很多视频开头并不干净地从“大家好”开始，而是先给观众一个场景：

- 产品上手镜头：开箱、撕膜、连接、测试；
- 体验高光：屏幕、游戏、降噪、眼镜、Mac mini 操作；
- VLOG 冲突：赶飞机、爬山、吃争议食物；
- 吐槽式悬念：为什么这么贵、为什么现在不要买、为什么我离不开它。

可迁移到用户的 AI / Obsidian / Hermes 内容：

```text
先不要解释系统架构。
先给一个“我真实遇到的问题/画面”：
- 我以为处理失败了，结果它已经写进 Obsidian；
- 我以为自动化最难的是模型，结果最难的是回报路径；
- 我以为整理文章会省时间，结果先被状态黑箱卡住。
```

### 4.2 任务声明：今天讲什么，为什么现在讲

常见骨架：

```text
大家好 / 哈喽大家
今天跟大家分享一下……
我现在已经用了 X 天 / X 个月 / 高强度用了一周
所以今天给大家总结一下……
```

不是正式论文开头，而是朋友式说明：我试过了，所以我来告诉你。

### 4.3 主体：首先 / 第二 / 第三，但每一点都落在体验

它的条目不是参数表，而是：

```text
首先，第一个最大的优点是……
  → 为什么这个点在真实使用里明显
  → 和旧款/竞品/旧方案比有什么差
  → 但是它也有一个限制
  → 所以对谁有用
```

尤其适合写工具/系统评测：

```text
第一，Mia 确实能把文章接住。
但真正影响体验的不是能不能接住，而是接住以后有没有告诉我放在哪里。
所以我现在更在意回报路径，而不是单次处理速度。
```

### 4.4 “但是”不是否定，是信任节点

ASR 里 `但是` 很密。它常见的作用有三类：

1. **卖点转体验**  
   `参数看起来很强，但是实际摸起来/用起来……`
2. **优点转限制**  
   `这个功能确实方便，但是如果你是 X 用户……`
3. **推荐转边界**  
   `现在可以推荐，但是它唯一的问题是……`

可迁移句式：

```text
它不是不能用，甚至已经挺好用了。
但是，如果你没有先把状态回报做好，它很容易变成黑箱。
```

### 4.5 “所以”负责把内容变成决策

`所以` 在样本里比“结论”更重要。它把一段体验压成行动建议：

```text
所以现在可以推荐。
所以现在先不要买。
所以如果你是老用户，可以升级。
所以如果你只是轻度使用，先等等。
```

写用户自己的内容时，应该强制每个段落都能接一句“所以”：

```text
它把文章写进 Obsidian 了。
所以，这套工作流的价值不是“AI读了文章”，而是“以后你还能找到它”。
```

## 5. 三种主要脚本模板

### 5.1 数码/工具评测型

适合：Hermes、Mia、Obsidian 插件、AI 工具、硬件、软件。

```text
1. 冷开场：一个真实画面/异常/高光
2. 任务声明：今天讲什么
3. 使用证据：用了多久 / 跑了几轮 / 什么场景
4. 先给倾向：推荐 / 不推荐 / 只推荐某些人
5. 首先：最大的体验变化
6. 第二：和旧方案相比的差异
7. 第三：容易被忽略但影响长期使用的点
8. 但是：最大缺点或边界
9. 所以：谁适合、谁先别上
10. 下一步：我后面还会继续测什么
```

### 5.2 买前必看 / 避坑型

适合：购买指南、升级建议、是否要换工具。

```text
1. 先制造决策场景：你现在可能正准备买/升级/切换
2. 说明为什么要现在讲：新品/版本/价格/评论区问题
3. 先承认它好在哪里，避免像纯黑稿
4. 但是：列出真正影响决策的 2–3 个缺点
5. 对不同人群分层：老用户 / 新用户 / 预算敏感 / 重度用户
6. 所以：现在买、等下一代、买旧款、还是完全不建议
```

迁移例子：

```text
别急着把所有文章都交给 AI 自动整理。
如果你还没有状态回报和人工复核位，自动化越强，黑箱越大。
```

### 5.3 VLOG / 生活化故事型

适合：把工具使用写得不那么“教程味”。

```text
1. 我为什么要做这件有点荒唐/麻烦的事
2. 中间发生了什么真实阻碍
3. 我当时怎么想，最好有一点自嘲
4. 结果到底有没有达到预期
5. 最后用一句朴素判断收束
```

迁移例子：

```text
我本来只是想让 Mia 帮我收几篇文章，结果一路查到了 Obsidian 路径、状态字段和回报机制。
这时候我才发现，AI 助理真正难的不是“会不会总结”，而是“你能不能放心地把信息交给它”。
```

## 6. 大耳朵式但去签名化：标题与开头模板

### 6.1 标题模板

不要直接复制“无敌了 / 必看 / feat. 长关键词 / 大耳朵TV”的完整签名。可用去签名化模板：

```text
真实用了 [时间/轮次] 后，我发现 [工具/系统] 最大的问题不是 [表面问题]，而是 [真实阻塞]
```

```text
[工具/系统] 到底值不值得上？我把优点、缺点和适合人群一次讲清
```

```text
别急着升级/自动化：如果你还没解决 [关键前置]，现在上反而更乱
```

```text
用了 [时间] 的 [产品/系统]，我现在只推荐 [人群A] 先上，其他人先等等
```

```text
[场景] 真实跑完后，我终于知道 [决策问题] 应该怎么选
```

### 6.2 开头模板

```text
我先给你看一个真实情况：
我把一批文章丢给 Mia 之后，本来以为它又卡住了。
结果回头一查，东西其实已经进了 Obsidian。
问题不是它没干活，而是它干完以后我不知道。
```

然后进入：

```text
所以今天我想讲一下，AI 信息流系统真正要先解决的，不是模型能力，而是反馈闭环。
```

### 6.3 缺点模板

```text
但是它不是完美的。
现在最大的问题是：[具体缺点]。
这个问题对 [人群/场景] 影响很大，因为 [原因]。
```

### 6.4 收束模板

```text
所以我的建议很简单：
如果你是 [A]，可以现在上。
如果你是 [B]，先等等。
如果你只是想 [C]，那它现在不是最优解。
```

## 7. 用户内容适配：AI / Obsidian / Hermes 写法

### 标题候选

```text
真实跑完 20 篇文章后，我发现 Mia 最大的问题不是不会总结，而是总结完你不知道去哪看
```

```text
别急着做全自动知识库：我用 Mia 接文章后，发现最该先补的是回报路径
```

```text
Hermes 助理到底值不值得长期用？我把优点、坑和适合人群一次讲清
```

```text
用了 Obsidian 接住微信文章后，我终于知道 AI 信息流系统该先解决什么
```

### 成稿示例

我先说一个真实情况：我把一批公众号文章丢给 Mia 后，第一反应其实不是“太智能了”，而是“它到底处理完没有”。

后来我去 Obsidian 里查，发现文章已经进去了。有路径、有标题，有些还被标到了后续要给 Atlas 或 Lisa 复核。

所以问题不是 Mia 没干活，而是它干完以后没有把结果清楚地告诉我。

首先，Mia 这套信息流的价值是能把内容接住。微信文章不再只是聊天记录里的一条链接，而是变成了本地 Markdown。这个变化很重要，因为它让“我看过”变成了“我以后还能找回来”。

第二，分类不是最难的，状态才是最难的。用户真正想知道的不是 AI 分了几个标签，而是：这篇文章现在在哪？有没有处理完？下一步归谁？我还要不要管？

第三，自动化如果没有收尾，就很容易变成黑箱。它可能已经做了很多事，但只要没有最后那句“已完成，路径在这里，下一步是这个”，用户体感上就等于没发生。

但是这套东西也不是完美的。它现在最需要补的不是更复杂的模型，而是一个稳定的反馈闭环：每次处理完，都要告诉我结果、位置、判断和负责人。

所以我的建议很简单：如果你也想搭 AI 信息流系统，不要第一步就追求全自动。先把四件事做稳：接住信息、写入本地、标明状态、回报路径。

自动化真正有用，不是因为它看起来高级，而是因为你不用担心信息掉进黑洞。

## 8. 禁用清单

为避免山寨感，不建议：

- 直接复制其标题原句；
- 同时堆满 `无敌了 / 必看 / feat. / 长关键词串 / 大耳朵TV`；
- 复刻“哈喽大家”式开场作为个人固定口癖；
- 在没有真实测试的情况下写“真实使用 X 天”；
- 只学强结论，不讲缺点；
- 把“但是/所以”当口头禅堆砌，而不是用于真实逻辑转折；
- 将本版 ASR 原文当精准逐字稿引用。

## 9. v0.6 队列

如果要继续升级，建议：

1. 用更高质量 ASR 模型重跑 6 条代表视频，修正 Whisper base 的错字；
2. 重试失败的 iOS26 正式版长视频，补系统更新型长视频完整节奏；
3. 抽 3 条完整结尾，验证“点赞三连/充电/下期再见”口播是否稳定；
4. 请 Claude Code / Claude Opus 再审一次 v0.5，重点看：是否过度模仿、证据分级是否准确、模板是否安全。
