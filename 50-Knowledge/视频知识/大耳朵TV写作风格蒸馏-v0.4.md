# 大耳朵TV写作风格蒸馏 v0.4

> **用途**：这不是逐字模仿手册，而是“结构借鉴版”。目标是把大耳朵TV的高点击评测表达拆成可迁移结构：强结论、真实使用、消费决策、朋友式说明、优缺点收束。  
> **边界**：尚未做 ASR，因此不能声称已经掌握完整口播节奏、转场、语速和镜头语言。本版只对标题、简介、选题包装和可迁移文案结构负责。

## 1. 一句话定位

大耳朵TV不是纯参数派数码号，而是：

> **苹果生态为主轴的真人朋友式消费决策频道：用真实使用时间建立可信度，用强结论标题制造点击，用优缺点和“谁该买/谁别买”完成决策收束。**

换成可迁移写作模型：

```text
我真的用了 / 试了 / 跑了一轮
  → 先给明确结论
  → 再讲具体体验
  → 必须讲缺点
  → 最后告诉读者该不该行动
```

## 3. 风格 DNA：可借鉴的五层结构

### 3.1 标题层：强结论 + 使用证据 + 决策词

大耳朵式标题的强处不是文艺，而是直接解决观众的消费焦虑：

- 到底值不值？
- 要不要升级？
- 能不能买？
- 和旧款/竞品比有什么变化？
- 我真实用了多久，结论变了吗？

可抽象为：

```text
[明确态度] + [真实使用证据] + [对象] + [关键变化/最大缺点] + [搜索关键词]
```

但不要复制整组招牌组合。尤其避免同时堆：`无敌了/必看/feat./关键词长串/大耳朵TV`，否则像山寨。

### 3.2 开头层：朋友式开场 + 先给结论

高频简介显示其开场很固定：

```text
嗨喽，大家好，今天分享……
```

可迁移原则不是照抄这句话，而是：

1. 低门槛、像朋友说话；
2. 直接说明今天讲什么；
3. 很快给结论，不绕玄学背景。

适合用户改写为：

```text
我先说结论：这套东西不是不能用，而是你必须知道它最容易卡在哪里。
```

### 3.3 主体层：体验不是参数，参数服务体验

大耳朵式评测不是“参数表朗读”，而是：

```text
功能 / 参数
  → 我在真实场景里怎么遇到它
  → 它解决了什么痛点
  → 它哪里没那么好
```

迁移到 AI/Hermes/Obsidian 内容，就是不要写：

> 这个系统有 worker、launchd、Obsidian、队列、分类器。

要写：

> 我把一堆文章丢进去后，最先发现的问题不是它不会处理，而是处理完以后我不知道它放哪了。

### 3.4 转折层：必须讲缺点，信任从缺点来

标题越强，正文越要有缺点。否则会像广告。

可复用句式：

```text
但它不是完美的。
真正让我犹豫的是……
如果你是 X，我不建议你现在上。
```

### 3.5 收束层：谁该做，谁先别做

最终不是“我觉得不错”，而是明确行动建议：

```text
如果你是 A，可以做。
如果你是 B，先等等。
如果你只想 C，那它不适合你。
```

这就是大耳朵式内容最值得借的部分：**把观看转化成决策**。

## 5. 文章/视频脚本骨架

```text
1. 朋友式开场
   今天我想讲一个我真实试过的问题：[对象/系统/产品]。

2. 先给结论
   我先说结论：[明确态度]。
   它最适合 [人群A]，但如果你是 [人群B]，我建议先等等。

3. 为什么我会试它
   我原本以为 [旧认知/期待]，但实际跑完后发现 [反差]。

4. 三个体验点
   第一，最明显的是 [体验点1]。
   不是因为参数好看，而是它在 [真实场景] 里解决了 [痛点]。

   第二，[体验点2]。
   和 [旧方案/竞品] 比，它最大的变化是 [差异]。

   第三，也是很多人会忽略的一点：[体验点3]。

5. 必须讲缺点
   但它不是完美的。最大的问题是 [缺点]。
   这个问题会影响 [某类人群/某个场景]。

6. 决策建议
   所以我的建议很简单：
   - 如果你是 [A]，可以做/可以买/可以升级；
   - 如果你是 [B]，先等等；
   - 如果你只想 [C]，它不是最佳选择。

7. 收尾
   后面我会继续跑 [下一步测试]，到时候再跟你们说它到底稳不稳。
```

## 6. 示例：用户 AI 助理内容的“大耳朵式但去签名化”版本

### 标题候选

```text
真实跑完一轮 Mia 文章处理后，我发现 AI 助理最大问题不是不会干活，而是干完你不知道
```

```text
别急着做全自动 AI 助理：我测试完 Mia 后，发现最该先补的是反馈闭环
```

```text
用 Obsidian 接住微信文章后，我终于知道 AI 信息流系统该先解决什么
```

### 成稿示例

我先说结论：AI 助理最容易让人失望的地方，不是它不会干活，而是它干完以后你不知道。

这次我把一批公众号文章丢给 Mia，本来以为中间又断了。后来回头查 Obsidian，才发现文章其实已经进去了：有标题、有路径、有分类，有些还被标到了后续要给 Atlas 或 Lisa 复核。

所以问题不是处理失败，而是反馈失败。

第一，信息真的被接住了。微信文章不再只是聊天记录里的一条链接，而是变成了 Obsidian 里的本地笔记。这个变化很关键，因为它让“我看到过”变成了“我以后还能找回来”。

第二，分类不是最难的，状态才是最难的。用户真正想知道的不是 AI 分了几个标签，而是：这篇文章现在在哪？有没有处理完？下一步归谁？我还要不要管？

第三，自动化如果没有收尾，就会变成黑箱。它可能已经做了很多事，但只要没有最后那句“已完成，路径在这里，下一步是这个”，用户体感上就等于没发生。

但这套东西也不是完美的。它现在最需要补的不是更复杂的模型，而是一个稳定的反馈闭环：每次处理完，都要告诉我结果、位置、判断和负责人。

所以我的建议很简单：如果你也想搭 AI 信息流系统，不要第一步就追求全自动。先把这四件事做稳：接住信息、写入本地、标明状态、回报路径。

自动化真正有用，不是因为它看起来高级，而是因为你不用再担心信息掉进黑洞。

## 8. 禁用清单

为避免变成山寨，不建议：

- 直接复用其具体标题原句；
- 同时堆叠 `无敌了 / 必看 / feat. / 长关键词串 / 大耳朵TV`；
- 在没有 ASR 的情况下声称掌握其完整口播节奏；
- 只学夸张标题，不讲缺点；
- 把“真实使用”写成空话，正文却没有真实场景。
