# Claude Code用了大半年，摸出5条原则（附我的CLAUDE.md从500行砍到47行的实录）

## Mia 核查

结论：**大方向是真的，适合作为 Skill / Agent 工作流素材；但文中的具体数字不要当硬规则。**

对照 Claude Code 官方文档：
- `CLAUDE.md` 作为项目/用户长期指令：✅ 官方支持。
- 不要把所有内容塞进 `CLAUDE.md`：✅ 官方建议控制长度，目标是 **每个 CLAUDE.md 低于 200 行**；文中“60 行/47 行”是作者自己的更激进实践。
- `.claude/rules/` 路径级规则：✅ 官方支持，适合大型项目按路径懒加载。
- Subagents 用于隔离上下文、拆分任务：✅ 官方支持。
- Hooks 跑测试/校验：✅ 官方支持；但“质量提升 2-3 倍”属于经验/转述，不是官方保证。
- “上下文 40% 开始变笨、60% 漏指令”：⚠️ 可作为经验提醒，但不是可验证的固定阈值。

可吸收原则：**别迷信提示词，应该把协作拆成记忆、计划、验证、分工、工具和 review。**

## Snapshot Text
去年，我刚开始用Claude Code。 

 跟很多朋友一样，上来就疯狂搜技巧。X上刷Boris Cherny的帖子，GitHub上翻84条最佳实践，群里见人就问"你有什么好用的招"。我把搜到的技巧攒了个文档，打算一条一条试。 

 越试越焦虑。 

 不是因为技巧不好用。是因为太多了。83条，14个分类。光读完就要两小时。想在项目里全用上？不可能。 

 后来我想明白了一件事：技巧永远学不完。底层原则就那么几条。 

 点个关注先: 

 过去一年，我从83条技巧里摸出了5条原则。这些原则不教你怎么做，教你为什么这么做。想明白为什么，很多技巧你自己就能推出来。 

 原则一：把上下文当内存管理 这是我踩过最大的坑。 

 刚开始用Claude Code的那段时间，我一个会话能干一整天。早上开一个窗口，一路干到天黑，从来不/compact，更不/clear。我觉得这就是"效率"——切来切去多麻烦。 

 然后Claude就开始犯低级错误。变量名写错，函数忘加参数，明明刚才说过的事情又问一遍。我以为模型出了问题，换了Sonnet换Opus，问题还在。 

 后来读到一条社区经验才明白：上下文过了四成，Claude开始变笨。过了六成，它会选择性漏掉指令，代码质量肉眼可见地往下掉。原因也简单——大模型能同时顾好的指令数量是有上限的，大概一百多条。Claude Code自己的系统指令就占掉了一大块。你堆再多规则，它根本装不下。 

 上下文用完不会突然崩。它会慢慢变笨。像内存泄漏——你以为还有空间，性能已经在降了。 

 我现在的习惯是50%就手动/compact。长任务主动拆成多段，每段新开一个会话。切换任务前让Claude写个交接总结，下一段知道从哪里接着干。 

 有人说Boris开10个终端15个会话一起跑，怎么不怕上下文炸？其实他用的就是这个原理——把一个大任务的上下文分散到多个独立会话里，每个会话只装一件事。 

 这个原则还可以推出一堆技巧： 

 回退比纠正更有效——走偏了按Esc Esc回滚，别在一个被污染的上下文里修修补补 子代理的价值在上下文隔离，不在炫技——20次文件读取和12次搜索的噪音留在子代理里，只把结论带回主会话 CLAUDE.md控制在60行以内——这是一张上下文预算表，不是文档库 原则二：给Claude一个验证自己工作的办法 有人问过Claude Code的创建者Boris，哪条建议最重要。他说：让Claude自己检查自己的工作。加了这个闭环，产出质量能翻两三倍。 

 我当时不太信。自己跑了一次对比。 

 第一次让Claude帮我写一个分子动力学模拟的数据处理脚本时，我走的老路子：它写完代码，我手动跑一遍，检查输出对不对。发现数值偏差，贴回去让它修。修完再跑，再看。来来回回四五轮，一个下午过去了。 

 后来我改了做法。让它写完脚本之后，自己跑一遍测试数据，自己对比输出和参考值，发现偏差自己改。我在旁边喝茶。平均下来，从写完到能用的轮次从四五轮降到了一两轮。 

 这里头的区别不在我少干了活。在反馈回路短了。Claude自己写→自己跑→自己看结果→自己改，整个闭环几分钟跑完一轮。而"我跑→我对结果→我描述偏差在哪→它理解→它修改"，每一步都在加延迟、加误读的可能。 

 这个原则推出来的技巧： 

 遇到报错，把错误信息贴给它，打一个字"修"。别指导。你排查的时间够它自己试三轮了 让Claude自己写验证脚本：处理完数据自动检查输出维度、NaN比例、数值范围 Claude写完代码后用Stop hook自动跑测试数据，结果对不上就让它自己修 但这事有个度。有些任务，验证条件天然清晰——数据处理管线，输入输出都是数值，对就是对错就是错。有些任务，验证条件本身就很难描述——"这个模型拟合得合不合理"，边界是模糊的。这种情况，人的判断还是少不了。 

 原则三：并行不是多开窗口 看到Boris说他同时跑5个本地终端加10个网页端，我第一反应是"这不就是多开吗，我也会"。 

 试了两天发现，多开5个窗口聊5个不同话题，结果一个都没聊透。 

 真正的并行不是多开窗口，是把任务拆成相互独立的单元。 

 举个例子。我重构一套化合物性质预测的代码管线时，开了3个会话。第一个读代码、画数据流图、设计重构方案。第二个写具体的重构代码。第三个——这个最关键——用一个独立的新会话，让Claude以不相关第三方的身份审查第二个会话改出来的代码。 

 三个会话干的是三件相互独立的事。读代码的不写代码，写代码的不审查自己写的代码，审查的不受前面任何上下文影响。这个效果比在一个会话里全部搞完强太多了。 

 Boris的5个终端也是类似的逻辑——5个Claude各干各的，不是凑一块做同一件事。关键是任务拆解得对：每个子任务有独立的输入输出，不需要频繁同步。 

 这个原则推出来的技巧： 

 代码审查时开多个子代理：一个查风格，一个翻历史找模式，一个找明显bug 用git worktree让多个Claude实例在不同分支上并行工作 Agent Teams——但要小心，每多一个agent上下文就多一份消耗。开5个并行就是5倍的量 原则四：规则是长出来的，不是写出来的——我的CLAUDE.md进化实录 前面三条原则聊的是"怎么用"。这一条聊"怎么建"——怎么搭一套让Claude越用越顺手的配置体系。 

 我拿自己的CLAUDE.md来展开。这是我最折腾、踩坑最多的部分。从快500行一路砍到47行，大概30个版本。每个版本后面都有一个"我当时真蠢"的时刻。 

 V1：500行的自信 刚发现CLAUDE.md这个机制的时候，我兴奋坏了。 

 当时的感觉是：啊，原来我可以把项目规则写进一个文件，Claude每次启动都会自动读？太爽了。 

 于是我开始疯狂往里塞东西。项目架构说明，写了三页。编码规范，从命名规则到注释格式到异常处理，一条不落。技术栈清单，RDKit、AutoDock、PyTorch每个依赖的版本号和用途都标上。还把各个数据处理脚本的调用约定、输入输出格式全部列了一遍。测试策略，单元测试怎么跑、用什么测试数据、精度阈值定多少。 

 我甚至还写了一个"项目历史"板块，记录了从立项到现在的所有重要技术决策。 

 写完一看，498行。我很满意。我想：这下Claude再也不会犯低级错误了。 

 实际情况是，Claude在接下来的三天里，犯的错误比之前还多。 

 V5：发现"选择性忽略" 错误主要集中在后半段规则。我在第300行写了一条"不允许直接改production分支的配置文件"，Claude转头就改了。我在第400行写了输出文件的命名规范，Claude完全没遵守。200行以后的规则基本等于白写。 

 一开始我以为是bug。后来才搞明白原理：大模型一次性能顾好的指令量是有上限的。Claude Code自己的系统指令已经占了不小的份额。你写300行规则，它能稳定遵守的不到一半。 

 更要命的是，规则越多，每条规则的权重就越低。Claude的处理方式有点像信息压缩——你交代了一百件事，它挑自认为最要紧的十来件记住。剩下的，当没看见。 

 这跟内存管理是一个道理。你有64G内存，开了8个虚拟机，每个分8G——跑不满的时候都正常，一满载就开始崩。 

 当时做了第一轮大砍。删掉了所有Claude读代码就能推断的信息。删掉了所有"以防万一"的冗余描述。从498行砍到180行。 

 但180行还是多。 

 V10：从"我预设它犯错"到"它真犯了再记" 砍到180行的时候，我停了一段时间。因为觉得再砍就没东西了。 

 直到有一次，Claude在一个数据处理项目里连续三次写错了自定义模块的import路径。第一次我手动改了，没在意。第二次我有点烦。第三次我打开CLAUDE.md，加了一行规则。 

 加了就加了，但加完之后我突然意识到一个问题：我原来那180行里，有至少100行是我"以为"Claude会犯的错。不是它真犯过的。是我凭经验预设的。 

 这两类规则的命中率天差地别。 

 预设的规则——"确保代码模块化"、"优先使用向量化操作"——Claude经常忽略。因为这些规则太模糊，而且跟它的默认行为本来就不矛盾。 

 "真犯过才记"的规则——"utils/目录下的自定义模块import用项目根路径，别用相对路径"——Claude几乎每次都能遵守。原因是精确的、有边界的、经历过失败验证的指令。 

 我开始把CLAUDE.md当成一个bug tracker，而不是设计文档。规则的新陈代谢变成：Claude犯错 → 我记录 → Claude下次遵守。后来我才知道，社区里流传的Boris团队工作方式，走的也是这个路子。 

 这一轮砍到了90行。 

 V18：.claude/rules/拆分 90行的时候，出现了新问题。 

 项目变大了。我的代码库里既有分子动力学模拟的C++代码，又有数据分析和绘图用的Python脚本，还有几个对接外部数据库的Rust工具。同一个CLAUDE.md既要管C++的编译规则，又要管Python的数据处理规范。很多时候，Claude在写Python脚本的时候，C++的构建规则也在上下文里占着位置。 

 这时候看到了.claude/rules/目录的机制。它是CLAUDE.md的升级版——你可以把规则拆成多个小文件，放在.claude/rules/里。关键是，可以用 paths: 这个YAML frontmatter来指定懒加载——只有Claude触碰到了匹配的文件路径，这条规则才被加载进上下文。 

 我把Python数据分析相关的规则放进一个文件，加 paths: ["**/*.py"] 。C++模拟代码相关的放进另一个，加 paths: ["**/*.cpp"] 。通用的项目规则留在CLAUDE.md主文件里。 

 这一拆分，每次会话的规则上下文从90行降到了30到40行。Claude写Python脚本的时候看不到C++规则，改C++的时候看不到Python规则。 

 但拆分带来了一个新坑：有些规则是跨边界的。比如"所有读写外部文件的路径都要用绝对路径"——Python脚本和C++代码都要读写文件，放在哪个规则文件里都不对。最后我把这种跨边界的规则放在CLAUDE.md主文件里，用 <important if="文件读写"> 标签包裹。 

 V25：发现Gotchas是最有价值的内容 到V25的时候，我的CLAUDE.md主文件稳定在50行左右。最密集的一行大概长这样： 

 用pandas读取SDF文件时，不要直接用read_csv，项目自己封装了mol_parser处理化合价（2025-12-14，浪费了40分钟排查） 
 这一行只有一句。但它包含了：具体场景（pandas读取SDF）、禁止行为（用read_csv）、原因（项目已有封装）、发生日期、犯错成本。信噪比极高。 

 这类内容在整个CLAUDE.md里占了大概30%。但这30%的内容贡献了80%的价值。因为它们是Claude最可能再次犯的错——具体场景下的惯性行为。 

 这跟Skill里建Gotchas的机制是同一个道理。每次Claude犯错，把这个失败模式记进去。时间一长，Gotchas变成整个配置里最有价值的部分。不是规范，是免疫系统。 

 V30：现在的47行 这是我现在CLAUDE.md的结构： 

 8行：项目构建命令和测试命令（Claude读代码推断不出来的操作信息） 12行：跨语言通用规则（文件路径规范、错误处理、数值精度约定） 3行：分支命名和PR规范 6行：MCP服务器配置 18行：Gotchas——Claude实际犯过的错和修正 总共47行。每条规则都是被验证过的。每条规则在Claude犯错之后才加上去的。没有一条是"以防万一"。 

 进化实录的几点教训 回头看这30个版本的折腾，有几点我当时不知道、但现在觉得太重要的事： 

 CLAUDE.md不是文档，是上下文预算表。  每多加一行，Claude就有多一行可能忽略。写规则之前先问自己：Claude读代码能推断出来吗？能就不写。 

 规则的价值跟它的精确程度成正比。  "写规范的代码"不如"测试文件统一放在tests/目录下，用test_前缀命名"。越模糊的规则越容易被忽略。 

 Gotchas是最值得维护的部分。  每一条Gotcha背后都是真实踩过的坑。它们不需要解释"为什么"，因为"为什么"就是那个坑本身。 

 别用CLAUDE.md代替Code Review。  很多人（包括以前的我）试图把Code Review的检查项全部写进CLAUDE.md。但静态检查该交给Linter、类型检查器、自动化测试。CLAUDE.md管的不该是代码质量本身，该是Claude特有的盲区。 

 这30版折腾下来，写CLAUDE.md花的时间确实比写代码多。但我现在觉得这不奇怪——写代码是让机器干活。写好CLAUDE.md是让AI把活干对。后者才是真正值钱的本事。 

 以上就是原则四的完整展开。它是我五条原则里花时间最多的一条。但值。CLAUDE.md就是你跟Claude之间的协作协议——协议写得干净，后面四件事（管上下文、设验证、拆并行、选工具）才跑得顺。 

 原则五：简单任务别上重武器 GitHub上有个整理Claude Code最佳实践的仓库，Star数快三万五了。里面收了10套主流工作流、152个子代理、194个命令、352个skill。 

 我看完第一反应是：全装上。 

 然后花了一整天配置其中最大最全的那套工作流。装完之后跑的第一个任务是改个变量名。本来是打一句话的事，硬生生走了Plan → Execute → Review三阶段，花了二十分钟。我坐那看Claude干活，比我自己改还慢。 

 后来我学乖了。给任务分了三个档位： 

 第一档，五分钟以内的活。直接说需求，不用Plan Mode，不用工作流，不用任何重型编排。原生Claude Code最快。 

 第二档，半小时到几小时的活。先Plan Mode讨论清楚方案，确认了再切Normal Mode执行。适当的Skill和Command辅助。 

 第三档，跨天跨模块的大活。上工作流，用Subagent拆分，开Worktree并行推进。 这个分级看起来简单，但执行起来有个坑：你很容易把二档任务误判成一档，或者三档误判成二档。我的规则是：如果做之前不确定大概要多久，直接升一档。 

 这个原则推出来的技巧： 

 Everything Claude Code适合全家桶党，Superpowers适合极简派——关键在于选对适合自己场景的 小任务不要建子代理，直接用主会话 /focus模式可以隐藏中间过程，只看最终结果——前提是你信得过模型 五条原则串起来 回头看我这一年用Claude Code的经历，从疯狂收集技巧，到只用五条原则做决策——效率确实涨了，但更重要的是，我终于不焦虑了。 

 新出一个技巧，我不用急着去试。我拿五条原则过一遍——它是管上下文的，还是管验证的，还是管并行的？归不到这五条里的，大概率是用不上的边角料。 

 如果五条原则非要拧成一句话： 把Claude当成一个团队成员——它资源有限、需要反馈、能并行工作、会从错误中学习、但不该被过度设计。 

 不是工具。是团队成员。这是Boris一直在做的事情，也是我这五条原则的来源。 

 工具你会用完就放下。团队成员你会不断改善协作方式。
