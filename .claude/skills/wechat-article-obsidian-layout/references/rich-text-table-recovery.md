# 富文本事件中的表格恢复

用于 `scripts/extract-public-wechat-html.py` 已按 DOM 顺序输出事件，但公众号用嵌套 `section` 模拟表格，导致单元格被扁平化的情况。

## 识别信号

把连续 `section` 文本视为“候选表格”，而不是默认当标题或列表。以下信号越多，表格判断越可靠：

- 开头出现 2–4 个短列名，如「Skill / 作用 / 值得学的地方」。
- 后续短文本可以按固定列数重复分组。
- 表格前一句有“分成三组”“可以这样拆”“结果如下”等引导。
- 表格后正文逐项解释同一批概念。
- 邻近信息图或截图展示相同名称、顺序或分组。

## 恢复步骤

1. 保留事件原顺序，定位连续的短 `section` 事件块。
2. 从前 2–4 项推断列头和列数；只在后续项能稳定整除或近似分组时建表。
3. 用邻近正文、代码名和信息图核对每行的列归属；图中可见的正式 Skill 名称可补回被样式节点遗漏的名称。
4. 不能可靠分列时保留为短列表，不要为了美观臆造表格。
5. 表格用于恢复原文结构；Mia 的补充解释仍放在摘要、图注或「可复用结论」，不要混入原文列值。

## 常见形态

- 对照表：`过去的用法 / 新的用法`，后续每两项一行。
- 三列表：`Skill / 作用 / 值得学的地方`，后续每三项一行。
- 审查表：`审查维度 / 实测发现 / 为什么重要`。
- 部门映射：`部门 / 可沉淀的 Skill`。

## 验证

- Markdown 表格行数与原始事件分组一致。
- 专有名词、数字和顺序与正文或图中证据一致。
- 没有把真正的章节标题吞进表格。
- 回读表格前后段落，确认文章叙事顺序连续。
