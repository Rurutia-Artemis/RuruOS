# 微信公众号公开页面类型与恢复规则

用于 `wechat-article-obsidian-layout` 在 public-only、无登录、无 Cookie 的边界内恢复正文结构与图片。

## 1. 普通富文本文章

识别信号：HTML 存在真正的 `#js_content` 或 `rich_media_content` 正文容器，且容器内有足量、连贯的正文事件。优先结合 `item_show_type: '0'`、正文字符数和段落层级判断。

**不要仅凭 `picture_page_info_list` 存在就切到图片卡片分支。** 普通富文本页也可能在脚本数据中携带该字段；只要 `item_show_type` 为普通页、`#js_content` 有完整正文和按 DOM 排列的正文图片，就仍按富文本处理。反过来，脚本/UI 中仅出现 `js_content` 字样也不算真正正文容器。

处理：

1. 从正文容器提取文本块，保留原始顺序。
2. 按 `p`、`h1-h4`、`section`、`blockquote`、`li` 恢复段落和标题。
3. 图片优先读取 `img[data-src]`，其次读取 `src`；按 DOM 顺序保存。
4. 重要图片本地化并紧邻对应段落，添加基于可见内容的简短图注。多张截图先做编号 contact sheet 总览，再只对文字密集或承载论点的图片逐张视觉读取；把终端输出、表格、时间段和关键数字补成紧邻图片的可搜索文本。
5. 安装命令、代码和终端输出改成 fenced code block；编号小标题映射为 Markdown 标题或高亮标题。
6. 元数据按公开 HTML 回退：标题优先 `og:title` / `msg_title`，署名可在 `nickname` 缺失时使用可信的 `author` meta；若二者实际上都是公众号名，来源卡可同时填写公众号与作者，但不要凭空推断个人作者。

完成条件：正文顺序完整、图片数与正文主图一致、关键图片信息可搜索、无微信热链、无广告/二维码/关注引导。

## 2. 图片卡片 / 轮播页

识别信号：没有真正的富文本正文，但存在 `item_show_type: '8'`、`share_content_page` 或 `picture_page_info_list`。注意：HTML 里可能出现 `js_content_container` 等脚本/UI 字符串，不能仅凭字符串 `js_content` 判成普通文章。

处理：

1. **不要相信 reader 的 `status: extracted` 就代表正文完整。** 若导出的“正文”主要是“微信扫一扫 / 使用小程序 / 取消允许 / 分享 / 收藏”等 UI 文案，而 meta description 有实质内容，应立即切换到图片卡片分支。
2. 以公开 meta description 作为文字来源；逐个解码字面量 `\\xHH` 转义（尤其 `\\x0a`、`\\x26quot;`），再做 HTML entity 解码。不要直接对整段中文使用 `unicode_escape`，以免破坏非 ASCII 字符。
3. 从原始公开 HTML 定位 `picture_page_info_list`，用**括号平衡扫描**取完整数组，而不是用贪婪正则猜数组结尾；再只读取数组一级对象的第一个 `cdn_url`，保持原顺序。
4. 忽略 `og:image` 封面、头像、二维码、页面 UI 资源，以及一级图片对象内部嵌套的 `watermark_info.cdn_url`。若用简单全局正则抓所有 `cdn_url`，很容易把水印和辅助资源误当正文图。
5. 批量下载一级主图后核验响应 Content-Type、文件签名、尺寸和非零大小。附件文件名使用当前日期前缀，并写入 live `AGENTS.md` 允许的域。
6. 图片卡片页的每张主图都承载正文时，逐张视觉读取是必要的；可以并行读取，但必须保留卡片顺序。把卡片网格转表格、命令转代码块、流程转标题/步骤，关键结论转正文，同时保留原图与图注。
7. 作者 metadata 为空时，可从所有卡片一致出现的署名恢复作者，但在“来源与备注”中说明署名来源；发布时间优先取页面明确的 `create_time: YYYY-MM-DD HH:MM`，不要从无关搜索参数里的时间戳猜测。

完成条件：公开摘要已解码；一级主图数、本地文件数和 Markdown 嵌入数三者相等；图片中的主要信息可全文搜索；无微信热链、封面、头像、水印或 UI 噪声被误当正文。

## 3. 公开请求被验证页拦截时的安全重试

首次公开请求若只返回“环境异常／去验证”，不要立即把文章判为不可提取，也不要进入登录、Cookie、扫码或验证码流程。可以在同一 public-only 边界内做一次确定性重试：

1. 改用移动端微信浏览器 User-Agent（iPhone + `MicroMessenger`）。
2. 给原始短链附加无权限含义的普通阅读参数，例如 `?scene=1`；不得添加账号凭证、Cookie 或私有 token。
3. 重新检查 `#js_content`、`item_show_type`、`picture_page_info_list`、标题 meta 和正文字符数。
4. 若重试仍是验证页，再按阻断处理；不要继续轮换代理、伪造登录态或自动完成验证。

这是“换公开页面形态再取一次”，不是绕过访问控制。若成功，应在正文备注中如实写明“公开 HTML，无登录、无 Cookie、无账号操作”。

## 4. 阻断与残缺

- 正文不足且图片列表也不可恢复：按 RuruOS `AGENTS.md` 路由到 `20-Reading/Queue/`。
- 只拿到截图时，明确来源是截图，不假装已抓到完整 HTML。
- 不使用登录态、Cookie、扫码或账号动作绕过微信限制。

## 5. 零正文图片的判定

正文图片数为零可以是合法结果，不要为了满足“保留图片”而把公众号封面、头像或二维码误存为正文附件。只有同时满足以下条件时，才可记录“正文无内嵌图片”：

- `item_show_type` 表明是普通富文本页，而不是图片卡片页；
- `#js_content` 已成功提取出完整、足量的正文；
- 正文容器中的 `img[data-src]` / `img[src]` 计数为零；
- `picture_page_info_list` / `picture_list_in_pictext` 也没有主图条目。

验证通过后，附件数和图片嵌入数都可以为 0；在“来源与备注”里说明封面未作为正文图片保存。

## 6. 无第三方解析库时的最小恢复方案

若环境没有 BeautifulSoup/lxml，不要为单篇文章临时污染全局 Python 环境。可用标准库 `html.parser.HTMLParser`：进入 `id="js_content"` 后按 DOM 顺序收集文本块、标题、列表、引用和 `img[data-src]`，离开容器即停止。输出至少包含：meta、正文字符数、正文图片列表、正文文本。完成后再人工恢复富排版层级并回读验证。

## 7. 验证清单

- 标题、作者、公众号、发布日期、原文链接可追溯。
- `dedup_key` / `source_url` 已查重。
- 普通页与图片卡片页已正确区分；验证页没有被误当正文。
- 主图全部本地化，路径符合当前 RuruOS `AGENTS.md`；若为零图片，已满足“零正文图片”判定条件。
- 图片事实有可搜索文字，不是只留图片。
- 文章正文与 Mia 摘要/可复用结论分开。
- 写入后回读，并以脚本检查 frontmatter 精确字段、章节、图片嵌入、缺失附件、微信热链和广告噪声。
